# Pivot Roadmap Reference

This document defines the negative-verdict playbook. It activates only when the verdict is "At Risk — Pivot Required" (Y5 = 30–49) or "Terminal — Sunset or Radical Pivot" (Y5 < 30).

The roadmap is **structured, quantified, and time-boxed**. No vague "consider pivoting to X" — every option has a target segment, a tech migration path, milestones, costs, and a kill switch.

---

## Trigger conditions

| Verdict | Y5 range | Roadmap depth |
|---------|----------|---------------|
| Durable | ≥ 70 | No pivot roadmap. Generate Defend & Extend plan (3–5 reinforcement moves). |
| Eroding | 50–69 | No pivot roadmap. Generate Reinforce or Niche plan (reinforcement + 2 niche-down options). |
| At Risk | 30–49 | **Full pivot roadmap** (this document). |
| Terminal | < 30 | **Radical pivot roadmap + sunset timeline** (this document, plus § 6). |

---

## Roadmap structure

The pivot roadmap has six sections, in this order:

### 1. Pivot trigger summary

One paragraph restating the verdict, the Y5 score, the dominant risk axis (which axis pulled the score down most), and the evidence behind it. This anchors the roadmap — every pivot option must address the dominant risk.

Example:
> Pivot triggered: Y5 score = 38.2 (At Risk). Dominant risk axis: AI Disruption Exposure (score 87/100). Primary driver: feature_automatability_pct = 92 — 92% of the product's current features can be replicated by a 2026-era LLM agent given existing tools. The pivot must either (a) move the product into a space where AI cannot follow, or (b) re-architect so the product is the AI's preferred distribution surface.

### 2. Pivot direction options (rank 2–3)

For each option, provide:

| Field | Description |
|-------|-------------|
| `pivot_name` | Short name, e.g., "Vertical-Deep Pivot" or "Distribution-Surface Pivot" |
| `target_segment` | Specific user segment (not "enterprise" — "compliance teams at mid-market financial services firms") |
| `target_segment_size` | Estimated TAM (USD) with citation |
| `value_prop` | One sentence: what uniquely valuable thing the pivot delivers |
| `why_ai_cant_follow` | The structural reason LLMs can't replicate this (proprietary data, regulatory license, physical-world, network effect, etc.) |
| `tech_migration_path` | Concrete: keep X, replace Y, build Z. Reference specific modules from the scan. |
| `migration_cost_estimate` | Engineer-months + infra cost (USD) |
| `time_to_revenue` | Months until first paying customer in the new direction |
| `expected_y5_moat_score` | Re-run the scoring model with the pivot's hypothetical inputs; report the projected Y5 |
| `probability_of_success` | LLM-judged, 0–100%, with rationale |
| `expected_value` | `probability_of_success × expected_y5_moat_score` — used for ranking |

The pivot options are generated by the LLM but **must** fall into one of five canonical pivot archetypes (see § 3 below). This constraint keeps the options grounded and prevents fantasy pivots.

### 3. Canonical pivot archetypes (pick from these)

Every pivot option must be classifiable as one (or a combination) of:

#### A. Vertical-Deep Pivot
Move from horizontal to a single vertical where domain expertise, regulatory compliance, or proprietary data creates a moat AI can't match.
- Example: Generic invoicing tool → invoicing for medical practices (HIPAA compliance, EHR integration).
- AI-can't-follow because: regulatory licensing, deep integration with vertical-specific systems, proprietary data accumulation over years.

#### B. Distribution-Surface Pivot
Stop being the destination; become the surface that AI agents call into. Expose your functionality as APIs/tools that LLMs prefer.
- Example: Weather app → weather API that AI assistants call natively.
- AI-can't-follow because: you ARE the AI's tool. The moat becomes API reliability, latency, data freshness, and brand trust with AI providers.

#### C. Data-Flywheel Pivot
Re-architect so the product accumulates proprietary data with every use, making the product better in a way AI alone can't replicate.
- Example: Generic CRM → CRM that benchmarks deal velocity against anonymized peer data.
- AI-can't-follow because: the value is in the aggregated dataset, not the workflow.

#### D. Physical-World Pivot
Add a physical-world component (hardware, in-person service, sensor network) that LLMs structurally cannot provide.
- Example: Inventory management software → inventory management + IoT sensors pre-installed at customer sites.
- AI-can't-follow because: atoms, not bits.

#### E. Trust / Compliance Pivot
Move into a regulated category where the moat is certification, audit trail, and legal liability — things LLMs cannot bear.
- Example: General-purpose analytics → SOC 2 Type II + HIPAA + FedRAMP-certified analytics for healthcare.
- AI-can't-follow because: regulatory bodies don't issue licenses to language models.

If the LLM cannot map a proposed pivot to one of these archetypes, drop it. This prevents vague "AI-powered" pivots that have no real moat.

### 4. Recommended pivot (single)

From the 2–3 options, pick the one with the highest `expected_value`. Provide:

- A 2-paragraph rationale referencing the dominant risk axis
- A 6-quarter (18-month) milestone table:

| Quarter | Milestone | Success criterion | Kill-switch trigger |
|---------|-----------|-------------------|---------------------|
| Q1 | Architecture spike | Prototype demonstrates the new moat mechanic | If prototype can't be built, abandon pivot |
| Q2 | First design partner signed | LOI from target segment | If no LOI by end of Q2, abandon |
| Q3 | Pilot deployment | ≥3 paying pilots | If pilots churn >50%, abandon |
| Q4 | GA launch | ≥10 paying customers | If CAC > 3× LTV, abandon |
| Q5 | Scale | ≥$100K ARR | If growth <20% MoM, abandon |
| Q6 | Moat validation | Re-run scoring model; Y5 ≥ 60 | If projected Y5 < 50, escalate to second pivot |

### 5. Resource plan

- Engineering headcount (FTE-months) by quarter
- Infra cost (USD) by quarter
- Marketing / GTM cost (USD) by quarter
- Total investment (USD) over 18 months
- Opportunity cost (USD) — what the current product would have earned during the pivot (use base-case projection)

### 6. Sunset plan (Terminal verdict only)

If the verdict is Terminal, also generate:

- A 12-month sunset timeline for the existing product
- Customer communication plan (when to notify, what to offer — refunds, data export, migration assistance)
- Data retention / deletion policy
- IP disposition (open-source? sell to competitor? archive?)
- Team reallocation plan (which engineers go to pivot, which are released)

## Defend & Extend Plan (positive verdicts)

For "Durable" verdicts, the skill produces a shorter plan instead:

3–5 specific moat-reinforcement moves, each with:
- The move (concrete action)
- Which axis it strengthens
- Cost (engineer-weeks or USD)
- Expected axis-score lift (estimate)
- Priority (P0 / P1 / P2)

For "Eroding" verdicts, also include 2 niche-down options (sub-segments where the product could dominate by focusing).

---

## Quality bar

The LLM must NOT:
- Propose a pivot without a `why_ai_cant_follow` rationale
- Propose a pivot that doesn't map to one of the five archetypes
- Set `probability_of_success` > 80% (anything that high is a sign of overconfidence)
- Set `expected_y5_moat_score` > 75 (the pivot is starting from a deficit; even a great pivot doesn't immediately hit "Durable")
- Skip the kill-switch column in the milestone table

These constraints are enforced by `generate_report.py` — the script will reject a pivot section that violates them and force a re-draft.
